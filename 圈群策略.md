根据终端云数据特征，构建多层实体和关系：

一层是玩家全生命周期实体，覆盖从初步触达到流失潜伏的 8 大模块、44 个核心维度；一层是玩家需求映射实体，把用户意图和产品物理特征接起来；另一层是游戏产品实体，把游戏拆成品牌、题材、美术、硬件、玩法、社交、付费、成长、多巴胺和运营活动等规格。与此同时，当前“完全具备”的维度大约只有 40%，大量关键深层行为仍然缺失。这个现实决定了 Agent 不能假设“所有特征都齐”，而必须具备规则优选、缺失容忍、代理推断和可解释输出的能力。

一、圈群到底解决什么问题

圈群是从1000万用户中选100万人出来，通过建模，可以按照属性和关系维度统一考虑，确定筛选规则；

它解决的不是“从 1000 万里直接找 100 万”的全部问题，而是更前面、更关键的一步：
对于任意一个新游戏或一个新 campaign，自动判断：这个游戏适合打哪些人，不适合打哪些人，应该优先启用哪些圈群规则，应该如何给后续模型提供召回池和解释。
所以它的职责是五件事：

读懂游戏

判断游戏满足哪些玩家需求

从玩家实体里选择匹配的圈群规则

输出可执行的人群包生成逻辑

给出解释、验证指标和兜底策略

一句话概括：
输入是游戏特征，输出是圈群策略。

二、解决方案：本体+Agent 形态：
1）主控 Agent（Orchestrator）
职责：

接收游戏信息

调度后面的子 Agent

汇总最终结果

输出规则包、候选人群包定义和解释

2）游戏理解 Agent（Game Parser）
职责：

读取游戏产品本体

识别这款游戏的核心特征

给游戏打“需求满足向量”

例如一款游戏如果是：

历史战争题材

强组织依赖

高博弈烈度

高身份外显

高满配投入

版本更新是赛季制

那它大概率满足的需求就是：

位次压制

身份隔离

利益分配

张力驱动

长期保值

荣誉确权

这一步其实就是把终端云文件里的“游戏产品本体”映射到“需求本体”。
3）规则选择 Agent（Rule Selector）
职责：

根据需求向量，从“玩家本体维度池”里挑选最合适的规则

给规则排优先级

判断哪些规则是必须、哪些是加分、哪些是排除

比如：

位次压制强 → 优先看历史投入、消费潜力、放弃竞争、博弈偏好

视听抚慰强 → 优先看画风倾向、题材偏好、视觉验证行为、终端性能

关系沉淀强 → 优先看社区身份、活跃时间段、组织依赖偏好、切断社交信号

4）证据与缺失处理 Agent（Evidence Agent）
职责：

判断规则所需数据是否具备

如果缺失，自动选择代理变量

给每条规则打置信度

标出哪些维度是“强规则”、哪些只是“弱证据”

这一步非常关键，因为一段时间内可能大量关键深层行为目前并不具备。比如综合繁忙度、加载留存、机制理解、付费动机、投身组织建设、分享战绩等都存在缺口。这个 Agent 需要知道：
没数据时用什么替代，不能因为缺字段就让整个规则失效。
5）人群包生成 Agent（Audience Builder）
职责：

把规则翻译成 SQL / 标签表达式 / 图谱查询

生成候选人群包

给出推荐触达渠道、素材方向、频控和排除条件

三、我们解决方案输入和输出应该是什么
输入
建议最少输入四类信息：
A. 游戏静态特征
直接来自游戏产品本体：

美术风格

文化题材

视觉表现力

交互逻辑

单局时长

反馈频率

博弈烈度

组织依赖度

身份外显度

满配投入

资产折旧率

版本更新量级等

B. 业务目标
这个很重要，因为不同目标圈群逻辑不一样：

装机量优先

高 D7 优先

首充优先

高 LTV 优先

大 R 获取优先

回流优先

裂变优先

C. 约束条件

预算

渠道

人群规模要求

是否允许高风险高成本人群

是否优先保护存量付费用户

D. 当前数据可用性

哪些玩家维度具备

哪些维度只有代理变量

哪些维度完全缺失

输出
Agent 不应该只输出一句“建议圈这批人”，而应该输出一整套结构化结果：

1. 游戏画像摘要
   这款游戏属于什么类型，主打什么体验
2. 需求满足向量
   例如：

位次压制：0.92

身份隔离：0.88

张力驱动：0.85

长期保值：0.77

本命寄托：0.25

3. 圈群规则优先级
   例如：

P0：历史投入高 + 消费潜力强 + 社区身份高

P1：策略偏好强 + 历史战争题材偏好 + 高博弈接受度

P2：活跃时间与组织玩法匹配 + 版本迭代关注强

排除：评分敏感极高且该游戏口碑分位一般的人群

4. 候选人群包定义
   例如输出成：

强匹配包：XXX

高潜扩展包:XXX

低成本探索包:XXX

不建议触达包:XXX

5. 规则解释
   每条规则为什么被选中，对应哪个需求、哪个游戏特征
6. 数据质量说明
   哪些规则是高可信，哪些规则依赖代理变量
7. 验证建议

A/B测试：测试用户分成两组，用当前解决方案和人工圈群，分类模型圈群做对比，收集比较：首充率，LTV，大 R 数等等（具体和终端云再商定）

四、核心逻辑：怎么“根据游戏特点选择合适规则”
第一步：把游戏特征转成“需求向量”

例如：

高博弈烈度 + 高满配投入 → 位次压制

高身份外显 + 买外观/特权驱动 → 身份隔离

强组织依赖 + 大型阵营 → 利益分配 / 集体背书

高视觉表现力 + 强美术风格 → 视听抚慰

高惊喜触发率 + 高反馈频率 → 惊喜反馈

低资产折旧 + 赛季制荣誉 → 长期保值 / 荣誉确权
这些映射终端云提供的需求特征里已经有明确基础。

第二步：把需求向量转成“玩家规则候选池”
举几个典型例子。
场景 A：重 SLG / 大世界战争 / 强联盟协作 / 大 R 深付费
应优先启用的规则：

历史投入高

消费潜力强

游戏分类偏好 = SLG/策略

题材偏好 = 历史/战争/权谋

社区身份高

活跃时段与团战窗口匹配

对版本迭代敏感

对组织协作玩法有历史偏好

排除规则：

极端碎片化用户

评分敏感极高但对该厂牌无信任的人

低设备兼容用户

场景 B：二次元 RPG / 强角色养成 / 视听驱动 / 中长线付费
优先规则：

画风倾向偏好 = 二次元 / 唯美

IP 偏好强

本命寄托倾向高

视觉验证行为强

题材偏好匹配

厂牌信仰强

首次登录后剧情沉浸倾向强

排除规则：

极端数值型、只看竞技压制的人

对长剧情耐受度低的人

场景 C：轻竞技 / 高反馈 / 碎片时间产品
优先规则：

综合繁忙度高

活跃时段碎片化

情境偏移动场景

物理顺滑需求高

压力宣泄需求高

单局短、反馈快偏好强

排除规则：

只吃重剧情、长沉浸体验的人

所以本质上，Rule Selector Agent 做的是：
先判断这款游戏“满足什么”，再反推“什么样的人会喜欢”。


第三步：从候选池里选最稳的规则组合
不是所有候选规则都能上。
Agent 要同时考虑四个维度：

区分度

覆盖率

数据可用性

业务可执行性

例如“投身组织建设”这个维度很强，但现在数据完全不具备（很难收集到），那它不能做强规则，只能降为未来增强项。反过来，“历史投入”“消费潜力”“题材偏好”“画风倾向”“活跃时间段”“评分敏感”这些具备度更高，就更适合成为现阶段主规则。

五、Agent 输出的规则结构：P0 / P1 / P2 / Exclusion
主要是规则分层设计
P0：核心硬规则
强约束，命中率最重要
例如：

游戏分类偏好匹配

题材偏好匹配

终端兼容

活跃时段匹配

历史投入 / 消费潜力达到阈值

P1：价值强化规则
提升高质量概率
例如：

社区身份高

厂牌信仰强

评分敏感与口碑分位匹配

对应需求得分高

P2：扩量规则
用于从强匹配扩展到高潜
例如：

泛兴趣相近

相似产品短期浏览过

轻度跨品类偏好相近

代理标签匹配

Exclusion：排除规则
排除明显不合适的，比直接选择合适的，更容易；
例如：

兼容性不达标

评分敏感极高但游戏口碑一般

高投诉高打扰风险人群

当前正被其他重点产品争抢的人群

已在目标产品里重度活跃的存量用户

六、Agent 的内部知识结构应该怎么建

1）规则知识库
保存：

每类游戏对应的优先规则模板

不同目标下的规则权重模板

常见排除条件

历史实验结果

比如：

“高博弈/高身份/高深付费游戏”的模板

“视听驱动/角色驱动游戏”的模板

“碎片时间/高反馈休闲竞技”的模板

2）映射知识库
保存三种映射：

游戏特征 → 需求

需求 → 玩家维度

玩家维度 → 可执行规则表达式
3）数据可用性知识库
保存：

每个维度当前是否具备

若不具备，代理变量是什么

当前版本该维度的置信度

例如：

综合繁忙度：不具备 → 用会话碎片化、登录时间分布、设备切换做代理

价值外显行为：不具备 → 用头像框点击、皮肤浏览、分享入口点击做代理

投身组织建设：不具备 → 用公会页访问、语音驻留、组队时长做代理

七、Agent 决策流程
Step 1：读取游戏特征
输入某个游戏的产品本体信息
Step 2：识别游戏主导体验
判断这款游戏更偏：

权力与统治

审美与角色

社交与组织

即时反馈与爽感

长期经营与保值

公平与规则信任

Step 3：生成需求向量
输出每种需求的强度分数
Step 4：匹配玩家规则候选池
从规则知识库拉出候选规则
Step 5：检查数据可用性
把规则拆成：

可直接执行

可代理执行

当前不可执行

Step 6：产出人群包方案
输出：

核心强匹配包

扩量包

排除包

Step 7：给后续系统提供结构化接口
传给：

召回系统

排序模型

素材生成系统

投放系统

Step 8：接收实验反馈
把线上效果回写到规则知识库，更新下一轮规则权重

八、agent 核心提示词设计

你是游戏圈群入口 Agent。
你的任务不是直接预测用户价值，而是根据目标游戏的产品特征、业务目标和可用数据，选择最合适的圈群规则。
你必须完成以下步骤：

识别游戏的核心体验与商业模式

将游戏特征映射到玩家需求向量

从玩家维度中选出最相关的规则，并区分 P0/P1/P2/Exclusion

检查数据可用性，缺失时启用代理变量

输出候选人群包定义、规则解释、风险点与验证建议

输出必须可执行、可解释，避免使用当前完全不可得的强依赖规则

然后工具侧给它：

游戏本体表查询

玩家标签表查询

数据可用性表查询

历史实验结果查询

人群包生成器
